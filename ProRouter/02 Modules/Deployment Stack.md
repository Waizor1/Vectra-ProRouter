## Constraints

- Контур все еще single-VPS и single-tenant.
- Controller feed build lane сейчас воспроизводим, но опирается на `system Go + manual ipk packaging + SDK usign/index tooling`, потому что `golang/host` в official OpenWrt `24.10.4` SDK на этом VPS оказался нестабилен.
- Smoke check пока не включает end-to-end `draft/apply`, rescue, и firmware validation.
- `/opt/vectra-prorouter` не является git checkout; безопасный prod sync здесь требует staging + `rsync --delete` release-slice flow с backup-директориями под `/opt/vectra-prorouter-backups/`, а не точечной дозаливки отдельных файлов, если в релизе были переезды или удаления исходников.
- Smoke check всё ещё не покрывает browser login screenshot, controlled rescue и firmware validation, но full `draft/apply` уже был отдельно пройден на live AX3000T.
- Firmware validation/artifact lane уже живёт в production contour; после реального sysupgrade package restore теперь имеет dry-run/read-only helper proof, но `-Apply` режим ещё не прогонялся как официальный post-flash восстановительный шаг.
- Controller/LuCI feed-signing lane is available again through the existing trusted key, but the safe production path is now explicit: build in `/tmp`, publish only the `.ipk` artifacts into the stable feed, sync metadata separately into PostgreSQL, and prefer direct `.ipk` install on routers when feed-name installs are brittle.
- The targeted `scp` lane remains acceptable only when the release is strictly an in-place edit to already-present files. If a slice adds, deletes, or moves web files, fall back to the staging + `rsync --delete` flow with backup directories instead of partial uploads.
- The published public helper archives are now structurally correct (`Vectra Install Helper.app` on macOS, platform launchers on Windows/Linux), but the macOS zips are still unsigned in the live contour: the packaging script supports `MACOS_SIGN_IDENTITY` plus `MACOS_NOTARY_PROFILE`, yet the current build workstation reports `0 valid identities found` from `security find-identity -v -p codesigning`, so Gatekeeper warnings remain expected until a real Apple Developer ID signing/notarization setup is added.
- 2026-04-10 production verification still shows that uploading fresh controller `.ipk` files and syncing PostgreSQL metadata is not sufficient to advance the public stable feed pointer: the direct artifact path and DB now already contain releases up to `0.1.12-r7`, but the signed `index.json` / `Packages` lane is still a separate close-out step. This contour therefore still depends on a reproducible signing/indexing lane, not just artifact copy.
- The production stack now intentionally avoids Docker service-name resolution for the hot path between `caddy -> web` and `web/db-backup -> postgres`: `docker-compose.yml` pins those peers onto `vectra_internal` with static IPs plus `extra_hosts`. This reduces the observed DNS flaps, but future deploys must preserve those IP assignments consistently or the mitigation will be lost.
- A 2026-04-15 production incident now also hardened the web deploy lane itself: a bad release sync touched `deploy/runtime` inside the non-git VPS deploy root, which made production come up against an empty Postgres data dir and temporarily emptied the panel until runtime was restored from backup. - The new background-push monitor is intentionally piggybacked on the existing single-node web process and is armed from `/api/health`, which is acceptable in the current `single-VPS + docker healthcheck every 30s` contour but is still an operational assumption that should be revisited if the web tier ever scales beyond one steady container.
- This production VPS currently mixes `UFW`, Docker-managed chains, and nftables-backed policy. Raw third-party firewall scripts such as `z4r` and `AntiScanner`, or interactive `PAM`/`sshd` installers like `distillium/motd`, are therefore not safe defaults here unless they are first replayed and diffed in an isolated staging host with the same network topology.

This is recoverable, but it is still a release-lane fragility until a cleaner mirror or cache policy is adopted.
