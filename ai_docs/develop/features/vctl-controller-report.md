# Контроллер нового поколения (vctl) — что сделано, чем лучше PassWall2, оптимизация под роутеры

> Простым языком, с примерами. Дата: 2026-06-01. Ветка `feat/vectra-controller-pro-autonomous`, PR #27.
> Статус: **код готов и проверен, на живой роутер ещё не выкатывали** (это отдельный гейт — см. конец).

---

## 1. Коротко: что это

Раньше на роутере работала связка **PassWall2** (куча Lua- и shell-скриптов) поверх Xray. Мы написали **свой контроллер `vctl`** — один маленький Go-бинарь, который **сам напрямую управляет Xray** и выкидывает PassWall2 целиком.

Аналогия: PassWall2 — это переводчик-посредник, который на каждый чих бегает, пересобирает конфиги пачкой скриптов и иногда падает от нехватки памяти. `vctl` — это один человек, который сам знает язык Xray и говорит с ним напрямую, без посредников.

В этот заход контроллер доведён до **полной поддержки Xray** + добавлена **защита ключей через HAPP CRYPT v5** (чтобы из приложения нельзя было вытащить VLESS-ключи).

## 2. Почему это лучше PassWall2 — с примерами

| Что | PassWall2 (как сейчас) | Наш `vctl` |
|---|---|---|
| **Из чего состоит** | Lua-интерпретатор + десятки shell-скриптов + куча пакетов | **Один статический Go-бинарь (~8 МБ)**, без зависимостей |
| **Что происходит при каждом применении конфига** | Форкаются `subscribe.lua`, `rule_update.lua`, `app.sh`, `nftables.sh`, dnsmasq-хелперы, `opkg status`… | Всё в одном процессе, **без форков на горячем пути** |
| **Память на слабом роутере** | Эти форки **реально вызывали OOM** (падение xray) на парке — `1111111111`, VagrandRouter, Kirill-MSK | Один процесс с лимитом памяти, не плодит форки |
| **Проверка живости нод** | `test.sh` **запускает ВТОРОЙ xray** на временном порту → удваивает память → это и был источник OOM | Лёгкие Go-пробы; заложен gRPC Observatory (метрики прямо из работающего xray, без второго процесса) |
| **Уважение к настройкам** | **Молча меняет** ваши значения: `fp=firefox` → `fingerprint=chrome` | **Ничего не меняет молча.** Поставили `firefox` — в Xray уйдёт `firefox`. Любая нормализация — только явно и с логом |
| **Безопасность фаервола** | Если правило кривое — трафик может утечь напрямую | **Commit-confirm авто-откат**: применили nftables с таймером; если роутер за N секунд не подтвердил, что связь жива, — правила **сами откатываются** (нельзя «потерять» роутер удалённо). Плюс опциональный **kill-switch** (трафик клиентов не утекает мимо прокси) |
| **Защита ключей в приложении** | Нет | **HAPP CRYPT v5** — ссылки-подписки шифруются так, что их расшифровывает только приложение Happ; пользователь не может посмотреть/скопировать/вытащить VLESS-ключи |
| **Целостность состояния** | — | Атомарная запись состояния (`.tmp + fsync + rename`), резервная `last-good` копия, восстановление после сбоя |

**Пример 1 — «почему падало».** На роутере с ~60 МБ свободной памяти PassWall2 при обновлении подписки форкал lua/shell, а `test.sh` поднимал второй xray для проверки — пик памяти, ядро убивало xray, прокси отваливался (это задокументированные инциденты на парке). `vctl` так не делает: один процесс, проверки лёгкие, тяжёлые операции **гейтятся по памяти** (не запустится, если RAM/overlay мало).

**Пример 2 — «почему ключ был не тот».** Оператор ставил `fp=firefox`, а на роутер уезжал `chrome` (PassWall2 молча подменял). На дешёвых анти-DPI это могло ломать обход. У нас — что поставили, то и уехало.

**Пример 3 — «нельзя потерять роутер».** Меняем фаервол удалённо. Если новые правила случайно рвут связь — раньше это означало «ехать к роутеру». Теперь deadman-таймер сам откатит правила за ~90 секунд, и роутер вернётся на связь сам.

## 3. HAPP CRYPT v5 — защита ключей (самая передовая часть)

**Задача:** чтобы из приложения нельзя было вытащить наши VLESS-ключи к серверам.

**Как сделано:** подписка (ссылка с ключами) шифруется в `happ://crypt5/...` — это формат приложения **Happ**. Расшифровать может **только само приложение Happ** (приватный ключ зашит в нём). Пользователь видит «крякозябры», не может посмотреть, отредактировать или поделиться серверами.

**Важная честная деталь:** алгоритм crypt5 у Happ **закрытый** (ключи зашиты в приложении и официально не раздаются). Поэтому:
- **crypt5** мы выпускаем через **официальный API Happ** (`crypto.happ.su`) — это даёт настоящие v5-ссылки, без лимита длины, и нам **не нужно хранить чужие закрытые ключи** (чисто с точки зрения лицензий).
- **crypt2/3/4** (RSA-4096) работают **полностью офлайн** на официальных открытых ключах (зашиты у нас, проверены).

**Как пользоваться:**
- С роутера/CLI: `vctl happ-crypt encrypt --version 5 --url '<ссылка-подписки>'` → выдаёт `happ://crypt5/...`.
- Из панели: процедура `happCrypt.encrypt({ url })` → `{ link }` (с кэшем, чтобы не гонять секрет в чужой сервис лишний раз).

Безопасность тут под особым контролем: ссылка отправляется только по HTTPS (запрет на «понижение» соединения даже через редирект), не пишется в логи, проверяется, что ответ — действительно `happ://crypt5/`.

## 4. Полная поддержка Xray

Контроллер умеет рендерить **весь рабочий конфиг Xray**, не только то, что использует парк сегодня:
- **Протоколы:** vless, vmess, trojan, shadowsocks, socks, http, hysteria2, wireguard, freedom (+ fragment/noise против DPI), blackhole, dns, loopback.
- **Транспорты:** tcp, ws, grpc, **http/2**, httpupgrade, kcp/mkcp, quic, xhttp, domainsocket.
- **Шифрование:** TLS (с uTLS-fingerprint, ALPN, pinning, ECH), REALITY (клиент и сервер), mux/xudp.
- **Маршрутизация:** правила + балансировщики + **Observatory/health** (нужно для leastPing/leastLoad — раньше балансировщик был «слепой»).
- **DNS:** все стратегии, FakeDNS, split-DNS.
- **Фаервол:** tproxy + fwmark + ip-rule, опциональный kill-switch.

## 5. Оптимизация под роутеры (конкретика)

Это написано под слабые Filogic-роутеры (Xiaomi AX3000T и т.п.), где мало RAM и overlay:

1. **Один бинарь, ноль форков на горячем пути** — убрали главный источник OOM (lua/shell/`opkg`-форки PassWall2).
2. **Лимит памяти на Xray:** обёртка `vctl-xray-wrapper` ставит `GOMEMLIMIT=80MiB`, `GOGC=30` (агрессивнее чистит память).
3. **Защита от OOM-киллера:** `oom_score_adj` — ядро в последнюю очередь убивает нужные процессы.
4. **Гейт по ресурсам:** тяжёлые задачи (применение конфига, обновление подписок/гео, установка пакетов) **не запускаются**, если RAM/overlay/tmp ниже порога — лучше отложить, чем уронить роутер.
5. **Мягкий лимит памяти → перезагрузка, а не убийство:** при превышении — аккуратный reload Xray, не kill.
6. **Ограниченные таймауты на под-процессы:** зависший хелпер не подвесит весь контроллер.
7. **stdlib-only:** контроллер не тащит внешних Go-зависимостей — меньше размер, меньше сюрпризов.
8. **Меньше места на overlay** после вывода PassWall2: остаётся только `xray + dnsmasq + vctl`.

## 8. Как потестить прямо сейчас (без железа)

- `cd router/vectra-controller-pro && go test ./...` — все тесты.
- `go run ./cmd/vctl render -config examples/p0-vless-reality.json` — посмотреть, какой Xray-конфиг рендерится.
- `go run ./cmd/vctl happ-crypt encrypt --version 5 --url 'https://<твоя-подписка>'` — получить защищённую `happ://crypt5/` ссылку (нужен интернет к crypto.happ.su).
- `go run ./cmd/vctl happ-crypt encrypt --version 4 --url 'https://<подписка>'` — офлайн-вариант (crypt4, RSA-4096).
